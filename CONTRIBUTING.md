# Contributing

This repo uses [skill-creator](https://github.com/anthropics/skills/tree/main/skills/skill-creator) from the Anthropic skills repo to build and validate every skill through a rigorous eval-driven process.

## How a Skill Gets Built

### 1. Draft the Skill

Write an initial `SKILL.md` based on domain expertise. Include:
- Clear rules and decision flowcharts
- Anti-patterns with BAD/GOOD code examples
- References in `references/` for deeper material the agent can consult

### 2. Write Evals

Create `evals/evals.json` with realistic coding prompts — the kind of task a developer would actually give an AI agent. Each eval includes:
- A **prompt** describing what to build
- **Assertions** — specific, objectively verifiable checks the output must pass

```json
{
  "skill_name": "my-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "Build a React component that...",
      "expected_output": "Brief description of correct approach",
      "assertions": [
        "Uses X pattern instead of Y anti-pattern",
        "Handles edge case Z correctly"
      ]
    }
  ]
}
```

Good assertions are specific and binary — either the code does the thing or it doesn't. Avoid subjective criteria like "clean code" or "well-structured".

### 3. Run Evals

For each eval prompt, run two subagents in parallel:
- **With skill** — the agent has access to the skill's `SKILL.md` and references
- **Baseline** — the agent works without the skill (or with the previous version if iterating)

This gives a direct comparison of what the skill actually changes.

### 4. Grade

Each output is graded against the assertions. Results are saved as `grading.json`:

```json
{
  "eval_id": 1,
  "variant": "with_skill",
  "expectations": [
    {"text": "Uses X pattern", "passed": true, "evidence": "Line 42 uses useMemo"},
    {"text": "Handles edge case Z", "passed": false, "evidence": "No handling for null input"}
  ]
}
```

### 5. Review & Iterate

Results are aggregated into a benchmark and opened in an HTML review viewer (generated by `skill-creator`'s `generate_review.py`). The viewer shows:
- Side-by-side outputs for each eval
- Pass/fail for each assertion
- Previous iteration outputs for comparison (iteration 2+)

Review the outputs, identify where the skill falls short, and refine. Each iteration gets its own workspace directory so nothing is lost.

### 6. Repeat

The loop continues until:
- Pass rates are consistently high across all evals
- The skill improves over baseline on most evals without regressing on others
- Human review confirms the outputs are genuinely good (not just technically passing)

## Iteration History

Each skill tracks its iteration count in the README table. For `better-use-effect`, 4 iterations refined the skill from a basic draft to one that reliably handles:
- Callback prop stabilization (ref pattern vs unnecessary stabilization)
- Non-primitive dependency traps (object props, local state objects, array narrowing)
- Derived state vs synced state
- Event handler vs effect boundary

## Adding a New Skill

1. Create `skills/<skill-name>/` with a `SKILL.md`
2. Add `references/` if the skill needs supporting material
3. Write at least 3-5 evals in `evals/evals.json`
4. Run through the eval loop (at least 2-3 iterations)
5. Update the root `README.md` table with the skill name, eval count, iteration count, and description
6. Open a PR

## Directory Structure

```
skills/
└── <skill-name>/
    ├── SKILL.md           # The skill definition (YAML frontmatter + markdown)
    ├── references/        # Supporting docs the agent reads as needed
    │   ├── topic-a.md
    │   └── topic-b.md
    └── evals/
        └── evals.json     # Evaluation test cases with assertions
```

## Guidelines

- **Keep SKILL.md under 500 lines.** Move detailed material into `references/`.
- **Explain the why.** Don't just say "never do X" — explain what goes wrong when you do.
- **Evals should be realistic.** Write prompts a real developer would actually type, not contrived toy examples.
- **Don't overfit.** If a fix only helps one specific eval but makes the skill more rigid, it's probably not worth it. Generalize from the feedback.
- **Track iterations.** Each iteration's workspace is preserved for comparison. Never delete previous iteration data.
